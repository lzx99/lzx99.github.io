    HTTP协议

    在TCP/IP四层网络结构（应用层，传输层TCP/UDP，网络层IP，网络接口层）中。OSI七层

    发送端在层与层间传输数据时，每经过一层都会被加上网络层的首部信息，接收端每经过一层都会删除一条当前网络层的首部信息。

    HTTP1/2是基于TCP/IP应用层的协议（HTTP3基于UDP），最初只用来传输HTML且只有GET方法，97年的1.1版本用了将近20年，2015年诞生了HTTP 2。

    • 无连接；即请求时连接，请求完释放连接，服务器收到请求，返回响应，服务器收到应答，断开连接。

      无连接设计之初目的是为了不占用网络资源，如果一个网页有包含许多请求（如很多图片），多次连接降低了网络效率，所以在HTTP 1.1时，产生了Connection: Keep-Alive，默认开启，使客户端和服务器在keep-alive规定时间内保持连接TCP连接，只有设置Connecttion：false时取消

    • 无状态：每个HTTP报文是独立的，服务器对于前面请求处理的信息，后面请求不能直接获取到前面请求中的信息。

      设计之初降低了服务器的压力，但是在动态Web时代，无状态特性阻碍了应用程序的实现。于是产生了记录状态的Cookie和Session。

      Cookie：会话保存在客户端，Cookie只能保存是字符串（4KB），切勿保存重要数据，默认浏览器关闭时清除session，除非服务端设置setMaxAge。一个cookie就是一个（String，String），使用request.getCookies()获得一个cookie数组，getName()获得cookie的name，getValue获得cookie的value

      Session：会话保存在服务器，在调用getSession时创建，判断cookie中有无sessionid，无就生成sessionID，定义一个会话状态，并将sessionid通过Cookie发送给浏览器。

      restful的前后端分离项目中一般会使用 access_token，因为前后端分离项目跨域，使用cookie里的sessionID不方便认证状态，这个token和session_id是一个意思。分离Cookie参考

    TCP和UDP

    HTTP报文解析

    Start line 起始行 — Header Filed 头部字段 — 空行 — Entity Body 实体主体

    起始行和头部字段构成HTTP报文头部（Headers）

    1. HTTP 请求报文

    请求行–HTTP头字段–空行–请求体

    浏览器只支持POST和GET这两种请求方法，实际上还有HEAD，TRACE，PATCH，PUSH等请求方法。一般纯查询请求可以使用GET，其他请求用POST。由于只能是ASCII编码，所以GET不能传输中文

    1. **GET请求：放在请求行(URL)**，会暴露在地址栏，有长度限制，会被主动Cache，只有默认编 码。
    2. POST请求：放在报文体，无长度限制，可以传输中文。POST请求也可以在请求行中写入数据，这时候类似GET请求，但一般没人会在form action或ajax url中加入这种处理

    2. HTTP 响应报文

    状态行–HTTP头–空行–响应体

    这个http响应的响应正文时纯文本，可以时html等等

    Etag，Vary: User-Agent，Cache-Control等都是常用的响应头设置

    3. HTTP Header

    HTTP Header中有四种Header Filed：

    通用头：请求头和响应头都可使用字段

    Cache-Control、Pragma、Date、Via

    Connection：keep-alive/close

    实体头：用于定义Entity Body（数据）的一些信息的头字段

    Location（重定向跳转就设置了这个）、

    Content-Type、Conetent-Encoding（响应内容的压缩）、

    Expire、Allow、Etag（资源标识符）、Last-Modify

    请求头：

    User-Agent、Host、Referer、Accept（希望接收的类型）

    响应头：

    Age（在代理服务器中缓存秒数）、Server（提供服务器信息）、Vary

    HTTP缓存

    HTTP缓存也称为Web缓存，主要就是客户端本身的浏览器本地缓存和代理服务器缓存（正向代理和反向代理）。只能缓存GET请求的内容，缓存css、js、HTML等静态资源

    缓存Cache-Control参考

    在HTTP1.0时代，pragma和Expires两个头字段可以设置缓存策略。

    到了HTTP1.1以后，又有Cache-Control设置缓存策略，HTML meta、动态网页中可以设置。

    ​ public：允许客户端和代理服务器缓存

    ​ private（GET请求默认）：允许客户端缓存

    ​ no-cache：允许客户端缓存，但每次快取需要由服务器请求缓存是否过期，未过期返回304。协商缓存

    ​ no-store：不允许缓存。

    ​ max-age=60：允许缓存保留60秒（自带no-cache），没有这个属性就会去查找expire。强缓存

    ​ must-revalidate

    这些设置缓存的头字段时通用头字段，当request和response都设置时，Apache或nginx等服务器可以设置听哪个request还是response的。

    • ping 域名：返回当前域名用默认DNS解析的IP，并显示时延迟

    • **nslookup 域名 [ DNS服务器 ] ** ：返回域名被默认或指定的DNS服务器解析成的所有IP地址（无优先级）

      dig ns com.

    DNS解析顺序及原理

    DNS即domain name system，就是将域名解析到IP的服务器，一共有13台IPV4根域名服务器。缓存到LDNS（递归服务器）的过程是迭代（没找到就下一级找，将上一级输出作为条件），LDNS到最终查询结果是递归查询（逐级查询到再逐级返回），从整个系统层级来说DNS是一种递归系统

    1. 浏览器的DNS缓存：chrome缓存60是，Safari缓存10S，chrome://net-internals/#dns可以查看设置和清楚

    2. 操作系统DNS缓存：

      windows默认固定保留一天（MaxCacheTTL=86400秒），ipconfig/displaydns查看。ipconfig/flushdns清除

      macOS中保留时间遵守由DNS服务器设置的TTL值，Time To Live，一般不会超过一周。

      macOS清除DNS缓存，sudo killall -HUP mDNSResponder

      Linux通过nscd实现DNS缓存，重启该服务将会刷新缓存

    3. hosts文件

    4. 向系统设定的LDNS（ISP DNS或公共DNS），先zone文件后dns服务器缓存（同样受TTL限制）

      ⚠️：操作系统不设定DNS时，实际会上抛给路由器，检查路由器有无DNS缓存，无缓存就会抛给ISP DNS

    5. LDNS向根域名服务器发送递归请求，选取最快的根域名服务器，在里面根据**.com—–NS记录，找到最快的负责解析.com的顶级域名服务器**，

    6. 然后在顶级域名服务器里根据qq.com—-NS记录，查找到负责解析这个qq.com次级域名的服务商，次级域名的域名解析一般就是权威服务器，如DNSPOD

    公共DNS

    为了防止DNS污染和运营商劫持一般会使用公共DNS，公共DNS也会被污染和劫持？

    但是一般的公共DNS影响CDN效率，权威DNS根据公共DNS出口的请求IP来判断用户本身的位置和运营商来分配CDN，而公共DNS节点不多的，位置很有可能有偏差，甚至导致给你分配一个国外IP的CDN。而运营商分配的CDN是最准确的。

    ECS协议（Edns-Client-Subnet，允许递归服务器发送用户的IP给权威服务器）能解决公共DNS的CDN分配不准确问题，但是需要递归服务器和权威服务器同时支持，目前并不是特别广泛

    DNS记录

    • A记录：最终主机地址
    • NS记录：域名—-NS（域名的解析服务器），NS只能是域名，如e.gtld-servers.net.这是一个.com域名的DNS
    • SOA：起始授权服务器，标记众多NS记录中的主服务器
    • CNAME（别名）：CNAME记录主要用于域名的内部跳转
    • MX（邮件交换器）

    DNS恶意解析

    2. 服务器负载均衡

    负载均衡（Load Balance）是一个概念，就是将负载（也就是任务）分摊到多个处理单元，降低处理压力。大型网站服务器集群负载均衡（SLB），一般通过DNS服务器解析承担第一级负载均衡，解析到负载均衡服务器进行第二轮负载均衡

    DNS负载均衡

    通过DNS解析到不同IP，达到访问集群服务器。智能DNS解析，可以解析成最近的服务器（CDN），不仅起到降低服务器压力，还能加快访问速度。

    优点是简单和成本低，只需要在DNS服务商

    缺点是，DNS负载均衡使用简单的轮询算法，不能根据服务器的运行状态和处理能力来完成负载均衡。

    负载均衡服务器

    通过负载均衡服务器在服务器集群中的选择服务器，负载均衡服务器接受普通TCP/UDP请求，与TCP三次握手也就是解析到报文的第4层后就发送到RS（Real server），接收http/ftp等请求，需要TCP握手三次之后，解析到第7层，才与RS建立连接。负载均衡服务器有三种传输模式：

    • 反向代理模式（proxy_pass）：代理服务器访问，服务器无法知道客户端的真实IP，只知道代理服务器的IP；反向代理，客户端无法知道服务器的真实IP，只知道反向代理服务器的IP（virtual IP）

    • 透传模式

    • 三角模式（IP欺骗）

    CDN

    CDN的全称是Content Delivery Network，即内容分发网络。将网站的静态内容发布到最接近用户的网络边缘，提高用户访问网站的响应速度。CDN有别于镜像，因为它比镜像更智能。CDN同样依靠智能DNS解析，通过CDN服务商的智能DNS将域名解析到最近的缓存服务器上，对于静态网站加速效果好，交互性网络不明显。

    ​ 当用到国外DNS服务商时，有可能会造成一定的国内网站，使用了 CDN 加速的国内网站的 DNS 会无法判断你的来源

    3. 网络线路优化

    线路优化主要是让用户以最快的速度访问到服务器，服务器线路如何搭建很重要，单线就是用户访问服务器只会是固定的联通或者电信，双线就是根据用户的IP走不同的运营商路由。

    • 单IP单线，一个服务器对应一个IP，这是最常见的最普通的网络方式
    • 双IP双线，一个服务器多IP（联通/电信/移动），用两个域名解析到这两个IP，或者配合智能DNS解析，DNS权威服务器根据递归DNS的出口IP代表用户IP线路，返回对应运营商的服务器IP，一般泛解析默认指向电信IP。
    • 普通单IP返程双线：在服务器的出口路由器配置路由表，达到返回数据时选择运营商线路，但用户发起请求达不到双线
    • BGP单IP双线：机房与电信联通达成合作协议，通过BGP协议选择多个运营商之间的最佳骨干路由器
    • CDN加速：通过CDN服务商的智能DNS将域名解析到最近的缓存服务器上，对于静态网站加速效果好，交互性网络不明显

    代理服务器

    代理软件

    基本国外的网站都走国外

    域名代理规则—–直接发送（防止在本地解析DNS）。配置尽量多的国外地址，防止落到IP判断，解析浪费时间

    域名直连规则—解析

    未知域名—一次解析—中国IP走直连，外国IP走代理。外国速度变慢

    普通代理

    代理客户端会接管系统代理，浏览器等软件会检测到系统代理来使用，此时浏览器等软件直接将HTTPS请求封装到sock5流量，不会使用DNS缓存

    socks5流量到达代理客户端后

    • 先匹配域名规则，走直连规则就由代理客户端DNS解析到IP，将socks5流量还原成TCP请求；域名走代理规则就本地不解析，用某种协议重组socks5流量发送其中的域名和其他数据到代理服务器。

    • 如果没有匹配到域名规则的域名，代理客户端解析到IP匹配IP规则，匹配到直连规则直接还原TCP连接；匹配到代理规则将域名通过某种协议重组socks5流量发送域名+其他数据给代理服务器，远程服务器用域名再次解析并发起链接。

    全局代理（透明代理）

    通过redir/tun2socks客户端截获系统所有DNS请求和TCP连接。然后经由redir客户端封装成socks5流量发送给代理客户端（或者redir自身建立TCP），代理客户端拿到socks5以后，同普通代理，按照域名规则和IP规则分流。有两种形式的截获。

    全局模式模式下就算是浏览器也必须要DNS解析拿到IP，才能发起TCP连接（HTTPS请求）。

    • 普通redir/tun2socks（真DNS解析），代理客户端截获操作系统的DNS请求，返回解析IP结果，软件发起TCP连接，代理客户端截获TCP连接，根据TCP请求中的IP反查出域名。

    • FakeIP形式的redir（假DNS解析）：代理客户端截获DNS请求不会解析，而是返回一个FakeIP，截获TCP连接时按照FakeIP反查出域名。相对于普通的redir，FakeIP会省去一次DNS解析，变成和直接设置代理差不多。

    DNS缓存的影响：可能通过浏览器或系统缓存获取IP而不会发起操作系统解析，DNS不会被截获，此时普通redir的域名规则代理失效，而Fake IP池会缓存之前的IP与域名的映射关系，不受DNS缓存的影响。

    Fancyss（路由器翻墙软件）使用的redir，surge的增强模式用的FakeIP，clash使用redir-host和FakeIP两种都有

    企业级局域网：AP- -Switch–Firewall –Router –Internet。

    AP负责认证

    路由器

    传统路由器是三层设备（网络层），在TCP/IP构建的层级上来说就是承担转发功能，两个端口之间不能处于同一个子网，端口之间本身就有直连路由功能。

    ⚠️：家庭路由器不是传统路由器，带路由+交换机+AP（无线交换机）的多用途设备

    1. IP+子网掩码按位与就 是子网，同一个子网可以通过ARP广播帧互相访问到，当IP不是子网内部时，数据包抛向默认网关（一般是路由器地址）。

    2. 路由器连接不同的子网，不同子网通信就要用到路由器，路由器收到IP数据包后，用目的地址和路由表（目的子网+子网掩码+下一跳地址）匹配，决定下一跳地址。

      路由器的直连路由会自动记录在路由表当中，同一个路由器的两个子网之间

    静态路由与动态路由

    静态路由协议比较简单，认为主动的在路由表中添加一条路由路径，所以重点讲解一下动态路由

    • 路由表的生成

    RIP动态路由协议，使得路由器之间互相发送自己的路由表信息。

    路由器接收到相邻路由器发送来的路由信息，会与自己路由表中的条目进行比较，如果路由表中已经有这条路由信息，路由器会比较新接收到的路由信息是否优先于现有的条目。如果优先于现有的条目，路由器会用新的路由信息替换原有的路由条目。

    • 协议的度量值与更新时间

    (1）RIP的度量值

    RIP路由协议使用跳数作为唯一的度量值。在RIP中，规定了跳数的最大值为15, 16跳视为不可达。因此，RIP路由协议不适用与大型的广域网。

    (2）RIP的更新时间

    RIP使用UDP协议，端口号是520。路由器启动后，平均每隔30s就从每个启动RIP协议的接口不断的发送出路由更新消息。

    • 水平分割（Split Horizon）

    网络收敛后，路由器仍然在以30s的时间间隔向外发送整个路由表。

    执行水平分割可以阻止路由环路的发生，水平分割的规则是：从一个接口学习到路由信息，不再从这个接口发送出去。水平分割不仅能够阻止路由环路的产生，同时也能减少路由更新信息占用的链路带宽资源。

    • 路由中毒，毒性逆转

    路由器将不可达网络度量值置为无穷大（也就是跳数为15以上)，并由该表项来引发一个路由中毒，中毒路由被发给邻居路由器以通知这条路径失效，其他路由器到收到一个路由中毒后发送一个毒性逆转，表示已经收到中毒路由信息。

    交换机

    hub和交换机是做拓展接口的，区别在于hub普通交换机是二层设备（链路层），二层设备之间，通过APR广播域进行通信

    ，而交换机划分VLAN，也就是划分广播域，即使处于同一子网也无法进行通信。局域网设计就是通过交换机+VLAN划分实现的。三层交换机具有路由功能

    NAT地址转化

    三层交换机、路由器都会进行NAT转化
